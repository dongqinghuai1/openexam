### 2.2 管理后台容器运行时问题修复

问题现象：

- 浏览器访问 `http://127.0.0.1:3000/` 后，登录请求 `/api/users/login` 返回 `404`。
- `favicon.ico` 返回 `404`。

原因定位：

- 管理后台前端请求使用相对路径 `/api`。
- Vite 开发环境可通过 dev server proxy 转发到 Django。
- 生产容器中使用 `nginx` 托管静态文件，但默认配置未代理 `/api`，因此请求被前端容器本身以静态资源方式处理并返回 `404`。

修复措施：

- 为前端容器增加自定义 `nginx.conf`。
- 将 `/api/` 代理到 `http://backend:8000/api/`。
- 对 `/favicon.ico` 返回 `204`，消除无意义 404 噪音。

验证结果：

- `curl http://127.0.0.1:3000/favicon.ico` 返回 `204`。
- `curl http://127.0.0.1:3000/api/users/` 返回后端 `403` 鉴权响应，说明前端容器到后端的 API 代理已生效。
- 浏览器控制台中的 `Unchecked runtime.lastError: Could not establish connection` 属于浏览器扩展注入噪音，不是项目代码错误。

### 2.3 默认管理员初始化与登录验证

问题现象：

- 登录页面不再返回 `404` 后，输入账号密码仍提示“用户名或密码错误”。

原因定位：

- 后端数据库中初始状态没有任何用户记录。
- 登录接口本身正常，但认证时查不到用户，因此始终返回 `401`。

修复措施：

- 新增 Django 管理命令 `python manage.py init_admin`。
- 在后端容器启动命令中自动执行管理员初始化。
- 默认初始化账号：`admin / admin123`。

验证结果：

- 容器内已确认存在管理员用户 `admin`，且具备 `is_superuser=True`、`is_staff=True`。
- 通过前端代理路径 `http://127.0.0.1:3000/api/users/login` 提交 `admin / admin123`，已成功返回 `token`、`refresh_token` 和用户信息。
- 直接访问后端接口 `http://127.0.0.1:8000/api/users/login` 同样登录成功。

第二轮结果：

- Docker Desktop 安装成功。

- Docker daemon 已成功启动，`docker info` 可返回服务端信息。

  1. 初始状态下 `docker-compose.yml` 中引用的两个 `Dockerfile` 不存在：
     - `04_后端开发/03_业务代码/Dockerfile`
     - `03_前端开发/01_管理后台/Dockerfile`
     上述问题已补齐。
  2. `compose up -d` 当前失败于镜像拉取阶段，无法访问 Docker Hub：
     - `redis:7-alpine`
     - `postgres:14`
     错误指向 `registry-1.docker.io:443` 连接失败。
  3. Docker Desktop 当前未配置 HTTPS 代理，设置文件中也未发现代理项。

最终结论：

- Docker CLI 已具备。
- Compose 文件语法已可解析。
- 前后端 `Dockerfile` 已补齐，Docker daemon 也已启动。
- 当前仍未达到容器可启动状态，阻塞点变为 Docker Hub 拉取受限，需要网络放通或配置 Docker Desktop 代理/镜像源。
