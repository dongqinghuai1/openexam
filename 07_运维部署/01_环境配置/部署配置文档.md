# 部署配置文档

这份文档分为两层：

- 当前仓库里已经落地并可直接复现的部署方式
- 如果后续要走生产化部署，可以在现有基础上扩展的目标形态

这样写的目的是避免把“当前真实可运行方案”和“未来生产架构设想”混在一起。

## 1. 当前已落地的部署方式

当前仓库根目录已经提供 `docker-compose.yml`，可直接启动以下服务：

- `eduadmin-postgres`
- `eduadmin-redis`
- `eduadmin-backend`
- `eduadmin-frontend`

对应关系如下：

| 服务名 | 作用 | 默认端口 |
|------|------|----------|
| `eduadmin-postgres` | PostgreSQL 数据库 | `5432` |
| `eduadmin-redis` | Redis 缓存 | `6379` |
| `eduadmin-backend` | Django + DRF + Gunicorn | `8000` |
| `eduadmin-frontend` | Vue 3 管理后台，Nginx 提供静态服务 | `3000` |

说明：

- 当前默认容器部署只覆盖管理后台，不包含教师端、学生端、家长端。
- 三端客户端目前是独立源码工程，需要按各自工程单独构建和部署。

## 2. 当前实际架构

```text
浏览器
  |
  +-- http://127.0.0.1:3000  -> 管理后台前端容器（Nginx）
  |
  +-- /api/* 反向代理到      -> Django 后端容器（Gunicorn, 8000）
                                |
                                +-- PostgreSQL（5432）
                                +-- Redis（6379）
```

这是当前仓库里已经被代码和验收文档共同验证过的最小可运行架构。

## 3. 当前目录与部署对应关系

| 代码路径 | 部署角色 |
|------|------|
| `03_前端开发/01_管理后台` | Docker 中的 `frontend` 镜像构建来源 |
| `04_后端开发/03_业务代码` | Docker 中的 `backend` 镜像构建来源 |
| `docker-compose.yml` | 本地/测试环境默认编排入口 |

## 4. 当前 Compose 行为说明

在项目根目录执行：

```bash
docker-compose up -d
```

后端容器启动时会自动执行：

```bash
python manage.py migrate --noinput
python manage.py init_subjects
python manage.py init_admin
python manage.py init_demo_users
gunicorn eduadmin.wsgi:application --bind 0.0.0.0:8000
```

这意味着首次启动通常会自动完成：

- 数据库迁移
- 默认科目初始化
- 默认管理员初始化
- 教师、学生、家长演示账号初始化

默认访问地址：

- 管理后台：`http://127.0.0.1:3000`
- 后端 API：`http://127.0.0.1:8000`
- PostgreSQL：`127.0.0.1:5432`
- Redis：`127.0.0.1:6379`

默认账号：

- 管理后台：`admin / admin123`
- 教师端：`teacher / teacher123`
- 学生端：`13800000002 / student123`
- 家长端：`13800000003 / parent123`

## 5. 当前环境变量规则

后端当前环境变量以 `04_后端开发/03_业务代码/eduadmin/settings.py` 为准。

核心规则：

- 未配置 `DB_ENGINE=postgres`、`DB_HOST`、`DB_NAME` 时，默认使用 SQLite。
- 配置数据库相关环境变量后，自动切换为 PostgreSQL。
- 未配置 `REDIS_URL` 时，缓存退回本地内存缓存。
- 当前配置里 `ALLOWED_HOSTS = ['*']`、`CORS_ALLOW_ALL_ORIGINS = True` 适合开发和演示，不适合直接用于生产环境。

## 6. 本地非 Docker 运行

### 后端

在 `04_后端开发/03_业务代码` 目录执行：

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py init_subjects
python manage.py init_admin
python manage.py init_demo_users
python manage.py runserver 8000
```

### 管理后台

在 `03_前端开发/01_管理后台` 目录执行：

```bash
npm install
npm run dev
```

### 三端客户端

教师端、学生端、家长端均为独立 Taro 工程，可分别执行：

```bash
npm install
npm run dev:h5
```

## 7. 生产化扩展示意

下面这部分是建议的目标形态，不代表仓库当前已经全部落地：

```text
公网流量
  |
  v
Nginx / 网关层（80/443, TLS, 反向代理）
  |
  +-- 管理后台静态资源
  +-- 三端 H5 静态资源
  +-- /api -> Django 应用集群
  +-- /media -> 对象存储或文件服务

Django 应用层
  |
  +-- PostgreSQL
  +-- Redis
  +-- Celery Worker / Beat
  +-- 第三方服务接入
```

如果要走这条路线，建议逐步补齐：

- 单独的反向代理层或网关层
- 生产环境专用 `.env` 与密钥管理
- HTTPS 证书与域名配置
- 对象存储、媒体文件策略
- Celery Worker / Beat 实际部署
- 日志采集、监控告警和备份恢复流程
- 真实第三方服务接入，而不是仅保留 Mock 链路

## 8. 建议的生产环境变量模板

以下是更贴近生产部署的环境变量示意，供后续扩展使用：

```bash
# Django
DEBUG=False
SECRET_KEY=replace-with-real-secret
JWT_SECRET_KEY=replace-with-real-jwt-secret

# Database
DB_ENGINE=postgres
DB_HOST=postgres
DB_PORT=5432
DB_NAME=eduadmin
DB_USER=postgres
DB_PASSWORD=replace-with-strong-password

# Redis
REDIS_URL=redis://redis:6379/0

# Celery
CELERY_BROKER_URL=redis://redis:6379/0
CELERY_RESULT_BACKEND=redis://redis:6379/0
```

说明：

- 当前代码虽然包含 Celery 配置项，但默认 Compose 并未启动独立 Celery 服务。
- 如果需要异步任务真正投入使用，应补充 Worker 和任务验证流程。

## 9. 运维注意事项

- 不要在生产环境继续使用默认账号和默认密码。
- 不要在生产环境保留 `ALLOWED_HOSTS = ['*']` 和全开放 CORS。
- 如果课堂、短信、支付等第三方能力要上线，必须先补齐真实密钥注入、回调校验和失败补偿机制。
- 当前文档中的三端客户端部署仅适合作为源码级说明，不代表已经有统一发布方案。

## 10. 推荐命令

启动全部默认服务：

```bash
docker-compose up -d
```

查看状态：

```bash
docker-compose ps
```

查看后端日志：

```bash
docker-compose logs -f backend
```

查看前端日志：

```bash
docker-compose logs -f frontend
```

重建前后端镜像：

```bash
docker-compose up -d --build frontend backend
```
