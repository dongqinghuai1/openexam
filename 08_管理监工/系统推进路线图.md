# 系统推进路线图

## 一、目标架构与现状差距

### 1. 已有基础

- 管理后台：Vue3 + Vite + Element Plus 已具备主页面结构，`npm run build` 可通过。
- 后端：Django + DRF 已具备 `users / edu / classroom / exam / finance` 五大业务域。
- 数据库：数据库设计文档较完整，领域划分清晰。
- 部署：已有 `docker-compose.yml` 和基础部署说明。

### 2. 当前主要差距

- Taro 三端仍未完成稳定构建，无法进入完整功能验收。
- Django 后端尚未在当前环境完成真实启动与接口 smoke test。
- 腾讯会议、录制、支付仍以 Mock 或半实现为主，未达到生产闭环。
- 测试目录已经开始沉淀报告，但缺少系统化执行结果。
- 当前架构中缺少通知中心、日志审计、系统配置、运营分析等完整落地能力。

## 二、阶段推进建议

### 阶段 1：可运行验证

目标：让前后端和课堂端都达到可启动、可构建、可 smoke test。

任务：

- 安装并打通 Python 运行环境
- 启动 Django 后端并执行 `manage.py check`
- 补齐或修复 Taro 三端构建链
- 完成核心接口 smoke test
- 校准 Docker 与环境变量

验收标准：

- 管理后台可访问
- Django API 可访问
- 教师端、学生端、家长端至少一个目标平台可构建

### 阶段 2：主流程联调

目标：打通报名、排课、上课、录屏、扣课、考试、成绩、订单的核心流程。

任务：

- 教务主流程联调
- 在线课堂 Token、会议、录制链路联调
- 考试提交流程、成绩记录联调
- 订单、支付、退款流程联调
- 补齐异常态和日志记录

验收标准：

- 教师可发起课堂，学生可进入课堂
- 上课后有记录、可回放、可扣课
- 学生可参加考试并生成成绩
- 订单与退款状态流转完整

### 阶段 3：专项能力补强

目标：增强课堂稳定性、报表能力、审计能力和运营能力。

任务：

- 腾讯会议/TRTC 真实接入
- 录制转码与文件存储打通
- 日志审计与通知中心建设
- 成绩分析报表与收入统计完善
- 性能、弱网、兼容性专项测试

### 阶段 4：上线准备

目标：具备发布和验收条件。

任务：

- 环境部署与监控告警
- 数据初始化和备份
- 安全加固
- 回归测试、验收测试
- 交付文档与培训材料

## 三、按角色拆解的推进任务

### 产品经理

- 完整输出业务流程、字段字典、状态流转、验收标准
- 对齐权限矩阵与角色边界
- 对课堂、考试、财务链路补详细 PRD

### 前端 A：管理后台

- 补齐后台真实联调页面
- 处理空态、错误态、分页与筛选
- 完善图表分析与财务页面

### 前端 B：课堂与学生端

- 优先修复 Taro 三端构建链
- 完成教师端、学生端、家长端核心页面联调
- 接入真实 RTC SDK 和回放页

### 后端 A：基础与教务

- 完成认证、权限、教务主流程接口校验
- 统一响应格式、分页、过滤规则
- 打通课时账户和排课链路

### 后端 B：课堂服务

- 打通会议、录制、回放、上课记录
- 接入真实云会议/RTC 服务
- 增加课堂异常状态处理

### 后端 C：考试财务

- 完成考试对象、提交流程、成绩计算
- 补齐支付、退款、收入统计
- 加强财务审计和报表导出

### 测试

- 先做功能 smoke test
- 再做主流程回归
- 最后做课堂专项和兼容性测试

### 运维

- 补齐 Python / Docker / 数据库 / Redis / 存储环境
- 完成部署、备份、监控和告警

## 四、当前建议的短期优先级

- Taro 三端构建问题
- Django 真实启动验证
- Docker 环境验证

P1：

- 教务主流程联调
- 考试提交流程和成绩生成验证
- 订单/退款状态流转验证

P2：

- 腾讯会议和录制真实接入
- 通知中心、日志审计、系统配置补全
